Worldsim Template — Repo & Scope Guide

What this repo actually does, how scenes & robots work, and what's in-scope for SiteForge.
HUD × Antim Labs · Worldsim Robotics track
One-sentence mental model: a scene is a physics world (a MuJoCo/USD/URDF file), HUD wraps it as an environment that exposes tools over MCP, an LLM agent drives those tools, and a task function reads the final sim state to compute a reward.
reset(scene)  →  drive the robot via tools  →  grade from sim state  →  reward + subscores

1 · How the repo flows

Five moving parts. The first two are what you edit; the rest is plumbing you mostly leave alone.

① The environment + tasks

environment/env.py · environment/tasks.py

Defines the Environment, spawns the sim, and declares tasks with @env.template. Each task: reset the scene → yield a prompt → read sim state → yield EvaluationResult(reward, subscores). This is where Phase 1 lives.

② The scenes

scenes/<scene_id>/

Each folder is a world: a scene.xml (MJCF) + metadata.json. Adding a scene = adding a folder here. We'll add scenes/construction-v1/.

③ The sim host

sim/host.py · sim/server.py

Runs Newton physics in its own process and serves the tool API (reset, step, render, get_object_state, move_gripper, …) over FastMCP. The viewer can own the main thread (WORLDSIM_VIEWER=1).

④ The robot control

sim/control.py · sim/franka_bridge.py

Franka-specific IK / end-effector control for the VLA path. Single-arm and hardcoded to Franka joint names. Not used for our multi-robot construction.

⑤ Agents / runners

examples/example_agent.py · run_vla.py · agents/

Ways to drive the env: an LLM over the tools, a scripted agent, or a VLA policy. You mostly use hud eval environment/tasks.py claude.

The HUD workflow around it

declare → taskset → runtime → run → learn

Run locally with LocalRuntime, scale on Modal, or hud deploy to the platform. Every rollout is traced with its reward breakdown.

2 · What formats a scene takes

On reset(scene_id) the sim looks inside scenes/<scene_id>/ for a scene file, in this priority order (or a Python builder):

FileFormatNotes for us
scene.xml / scene.mjcfMuJoCo MJCFUse this. The bundled tabletop-v1 is MJCF; full names/cameras/rendering supported. Easiest to hand-edit.
scene.usd / scene.usdaOpenUSDSupported by Newton import; what Antim's Gizmo can export. Good for the pretty visual layer later.
scene.urdfURDFRobot description format. Imports geometry + joints, but no MuJoCo names/cameras layer.
build_scene.pyPythonA build() function returning a Newton model. Most flexible, most code.

Only .xml/.mjcf get the extra MuJoCo model loaded for human-readable body/joint/camera names + rendering — which is exactly what the grading tools (get_object_state, etc.) read. So for hand-authored work, stick to MJCF.

3 · How to add a new scene

  1. 1. Create a folder: scenes/construction-v1/
  2. 2. Add scene.xml (MJCF): floor, a build platform, named blocks as free bodies, and your manipulator(s). Easiest start: copy scenes/tabletop-v1/scene.xml and swap the contents.
  3. 3. Add metadata.json: scene_id, description, the objects (with names + initial positions), and any gripper/actuator descriptions. This is what the agent prompt and your grader reference.
  4. 4. Reference it from a task: await sim_tools.call("reset", scene_id="construction-v1") inside an @env.template.
  5. 5. Smoke test: python scripts/check_setup.py then hud eval environment/tasks.py claude --group 1. Watch it with WORLDSIM_VIEWER=1.
Minimum viable scene A floor geom, a fixed build pad, two free-body blocks (base_block, roof_block) each with a <freejoint>, and one floating gripper (copy the gripper block verbatim from tabletop-v1). That alone is enough for the Phase 1 build-house task.

4 · How to add a robot (and the humanoid question)

A robot is just more bodies/joints/actuators in the MJCF. The Franka scene shows the clean pattern — it pulls a whole robot model in with a single include:

<!-- scenes/franka-libero-v1/scene.xml -->
<include file="franka_emika_panda/panda.xml"/>   <!-- merges the robot's bodies, assets, actuators -->

So to add an embodiment (a Unitree/Spot-style URDF, a humanoid MJCF, etc.):

  1. 1. Drop the robot's model file + its mesh assets/ into the scene folder (e.g. scenes/construction-v1/unitree_h1/).
  2. 2. <include> it from scene.xml, or paste its <body> tree into the worldbody. Give it a base pos so robots don't overlap.
  3. 3. Make sure every actuator has a unique name (prefix per robot, e.g. h1_*, crane_*).
Reality check — robot control is the hard part, not the geometry Importing a humanoid's geometry is easy. Making it act is not. The repo's only motion controller (sim/control.py) is a Franka-specific IK that auto-activates only when it finds the exact Franka joint names (joint1..7, finger_joint1/2, actuator1..8, eef site). A dropped-in Unitree humanoid would be inert articulated geometry — there is no locomotion or whole-body controller, and you'd be commanding raw joint torques via step(action). Building a stable legged-locomotion controller is days of work, not hours.
Reality check — the high-level gripper tools are single-robot move_gripper, open_gripper, close_gripper, rotate_gripper are hardcoded to the tabletop gripper: body gripper_base, joints finger_left_slide/finger_right_slide/gripper_yaw, and actuator indices ctrl[0..5]. They control whichever actuators sit at those indices. Add a second robot and these tools no longer mean what you think. For multi-robot you drive the full actuator vector with raw step(action), or you author per-robot tools.

5 · What that means for SiteForge (our scope)

Do NOT do in 10 hours
  • Real Spot / Unitree humanoids with legged locomotion or whole-body control.
  • N independently-IK'd arms reusing the Franka VLA control path.
  • Trusting move_gripper & friends with more than one manipulator in the scene.
DO this instead
  • Phase 1: one MJCF scene construction-v1 = floor + build pad + a few free-body blocks + the proven floating gripper. Task build-house: stack base_block then roof_block onto the pad. Reward = per-block closeness to the target house pose + a stability check after settle. Reuses the exact move-object / pick-object mechanics → low risk.
  • Phase 2 (fleet): model 3 embodiments as floating manipulators with capability profiles (crane = heavy/slow, humanoid = dexterous, wheeled = fast/light). Theme via metadata + per-robot capability limits enforced in a thin tool layer; drive via step(action) over the full actuator vector. The narrative ("different embodiments, balance fleet use") holds without locomotion.
  • Fleet fallback (if multi-gripper physics fights you, time-box 60 min): keep ONE physics arm, make the fleet symbolic — a scheduler assigns blocks to named robots with capability constraints, physics executes sequentially, and reward credits utilization + makespan as if parallel. Whole story preserved, zero bridge rewrite.

The reward we're measuring (the "real task")

reward = 0.40 * structural_completion   # each block within tolerance of its target pose
       + 0.20 * fleet_utilization        # work spread across robots, not one doing everything
       + 0.15 * makespan_efficiency      # parallel build beats sequential
       + 0.15 * safety_no_collision      # no two manipulators in one cell; nothing knocked over
       + 0.10 * stability                # structure still standing after a settle step

Same partial-credit-from-sim-state pattern the template already uses for move-object, just composed over multiple blocks and robots. That composability is why it's tractable.
